## 1. Предварительные требования

### 1.1. Версия платформы и конфигурации

| Компонент | Требование |
|-----------|------------|
| Платформа 1С:Предприятие | 8.3.16 и выше |
| Конфигурация | 1С:Управление торговлей, редакция 10.2 |
| Режим клиента | Толстый клиент (обычные формы) |

> **Важно:** Расширение не поддерживает тонкий клиент и управляемые формы. Убедитесь, что пользователи работают в режиме толстого клиента.

### 1.2. Права доступа

Для установки расширения требуются:
- Роль **Администратор** в информационной базе 1С
- Доступ к режиму **Конфигуратор**
- Право на изменение конфигурации базы данных

### 1.3. Доступ к сервису NiFi

Перед установкой уточните у администратора инфраструктуры:

| Параметр | Где получить | Пример значения |
|----------|-------------|-----------------|
| URL NiFi endpoint | Администратор инфраструктуры | `https://nifi.internal:8443/api/v1/events` |
| API-ключ | Администратор инфраструктуры | `eyJhbGciOi...` |
| Сетевой доступ с сервера 1С | Проверить ping/telnet | Порт 8443 открыт |

> **Проверьте сетевой доступ заранее.** Если сервер 1С не видит NiFi — расширение не заработает. Убедитесь, что маршрут между серверами открыт и порт не заблокирован файрволом.

### 1.4. Наличие файла расширения

Файл расширения имеет расширение `.cfe`. Получите актуальную версию у разработчика или из корпоративного репозитория.

## 2. Установка расширения

### 2.1. Открытие конфигуратора

1. Запустите 1С:Предприятие.
2. Выберите информационную базу.
3. В окне запуска выберите **Конфигуратор** и нажмите **ОК**.

### 2.2. Добавление расширения

1. В конфигураторе откройте меню **Конфигурация → Расширения конфигурации**.
2. В открывшемся окне нажмите кнопку **Добавить** (знак «+» или пункт меню **Добавить расширение конфигурации**).
3. В появившемся диалоге выберите **Загрузить расширение из файла**.
4. Укажите путь к файлу `.cfe` и нажмите **Открыть**.
5. Дождитесь загрузки. Расширение появится в списке.

### 2.3. Активация расширения

1. Убедитесь, что в строке расширения стоит галочка в колонке **Активно**.
2. Если галочка не стоит — щёлкните по ней, чтобы включить.

### 2.4. Обновление базы данных

1. После добавления расширения выберите меню **Конфигурация → Обновить конфигурацию базы данных** (или нажмите **F7**).
2. В диалоге подтверждения нажмите **Принять**.
3. Дождитесь завершения обновления.
4. Закройте конфигуратор.

> **Рекомендуется** выполнять установку в нерабочее время или при минимальном числе активных пользователей. Обновление конфигурации базы данных кратковременно блокирует работу системы.

## 3. Настройка параметров

После установки расширения необходимо заполнить константы подключения. Выполняется в режиме предприятия с правами администратора.

### 3.1. Путь к настройкам

1. Запустите базу в режиме **1С:Предприятие** (не конфигуратор).
2. Откройте меню **Сервис → Прочие настройки → Интеграция: Сервис рентабельности**.

> Если пункт меню не отображается — убедитесь, что расширение активно и база данных была обновлена (шаг 2.4).

### 3.2. Параметры подключения

Заполните следующие поля:

| Параметр | Описание | Рекомендуемое значение |
|----------|----------|----------------------|
| **URL NiFi** | Адрес точки приёма событий | `https://nifi.internal:8443/api/v1/events` |
| **API-ключ** | Ключ аутентификации | Получить у администратора инфраструктуры |
| **Таймаут запроса (сек)** | Максимальное время ожидания ответа от NiFi | `5` |
| **Интервал повтора (сек)** | Как часто повторять недоставленные сообщения | `60` |

### 3.3. Правила заполнения

- **URL NiFi** — вводить полный адрес, включая протокол (`https://`) и путь. Без завершающего слеша.
- **API-ключ** — копировать точно, без пробелов в начале и конце строки.
- **Таймаут** — не устанавливать меньше `3` секунд во избежание ложных ошибок при нагрузке на сеть. При нестабильной связи увеличить до `10`.
- **Интервал повтора** — определяет, как часто проверяется очередь недоставленных сообщений. Значение `60` подходит для большинства установок.

### 3.4. Сохранение настроек

После заполнения нажмите **Записать и закрыть**. Настройки применяются немедленно, перезапуск базы не требуется.

## 4. Настройка регламентных заданий

Расширение использует два регламентных задания. Оба необходимо настроить и включить.

### 4.1. Задание «Отправка очереди событий»

Это задание проверяет и повторно отправляет сообщения, которые не удалось доставить с первого раза.

1. Откройте **Администрирование → Регламентные и фоновые задания**.
2. Найдите задание **Интеграция: Отправка очереди событий**.
3. Дважды щёлкните по заданию для редактирования.
4. Установите параметры:
   - **Расписание:** каждые `60 секунд`
   - **Разрешить** — поставьте галочку
5. Нажмите **ОК**.

### 4.2. Задание «Получение входящих команд»

Это задание опрашивает сервис рентабельности на наличие новых команд (результаты согласования, санкции, блокировки).

1. Найдите задание **Интеграция: Получение входящих команд**.
2. Дважды щёлкните по заданию для редактирования.
3. Установите параметры:
   - **Расписание:** каждые `30 секунд`
   - **Разрешить** — поставьте галочку
4. Нажмите **ОК**.

### 4.3. Проверка активности заданий

После сохранения убедитесь, что оба задания:
- Отображаются в списке регламентных заданий
- В колонке **Разрешено** стоит «Да»
- Колонка **Последний запуск** обновляется (признак того, что задание выполняется)

> **Если задания не запускаются** — проверьте, что в свойствах информационной базы включены фоновые задания. Для файловой базы они выполняются только при открытом сеансе с правами администратора.

## 5. Проверка работоспособности

После установки и настройки выполните проверку по следующему чеклисту.

### Чеклист проверки

- [ ] **Расширение установлено и активно**
  Конфигуратор → Расширения конфигурации: расширение есть в списке, галочка «Активно» стоит.

- [ ] **Константы заполнены**
  Сервис → Прочие настройки → Интеграция: поля URL и API-ключ заполнены, не пустые.

- [ ] **HTTP-запрос доходит до NiFi**
  В настройках интеграции нажмите кнопку **Проверить соединение**. Должно появиться сообщение «Соединение установлено успешно».

- [ ] **Подписки срабатывают при проведении документов**
  Создайте тестовый заказ покупателя и запишите его. В журнале событий интеграции (**Сервис → Журнал интеграции**) должна появиться запись об отправке события `order.created`.

- [ ] **Очередь повторной отправки работает**
  Тест устойчивости: временно укажите неверный URL NiFi → создайте заказ → верните правильный URL → через 60 секунд в журнале интеграции должна появиться запись об успешной повторной отправке.

- [ ] **Входящие команды обрабатываются**
  Проверьте с администратором сервиса рентабельности: отправьте тестовую команду результата согласования → в 1С у соответствующего документа должен измениться статус.

### Журнал интеграции

Для мониторинга работы расширения используйте встроенный журнал:

**Сервис → Журнал интеграции: Сервис рентабельности**

| Колонка | Что означает |
|---------|-------------|
| Дата и время | Когда произошло событие |
| Тип | Исходящее / Входящее |
| Событие | Название (например, `order.created`) |
| Статус | Отправлено / Ошибка / В очереди |
| Документ | Ссылка на документ-источник |
| Ошибка | Текст ошибки (если статус «Ошибка») |

## 6. Устранение неполадок

### «Не удалось подключиться к NiFi»

**Симптом:** Кнопка «Проверить соединение» возвращает ошибку. В журнале — «Ошибка HTTP: Connection refused» или «Unable to connect».

**Причины и решения:**

| Причина | Как проверить | Решение |
|---------|--------------|---------|
| Неверный URL | Скопируйте URL в браузер на сервере 1С | Исправить URL в настройках |
| Закрыт порт файрволом | `telnet nifi.internal 8443` | Открыть порт на файрволе |
| Сервер NiFi не запущен | Обратиться к администратору инфраструктуры | Запустить NiFi |
| Нет сетевого маршрута | `ping nifi.internal` | Настроить маршрутизацию |

---

### «401 Unauthorized»

**Симптом:** Соединение устанавливается, но ответ — ошибка авторизации.

**Решения:**
1. Проверьте API-ключ: скопировать заново из источника, убедиться в отсутствии лишних пробелов.
2. Уточните у администратора инфраструктуры, не истёк ли срок действия ключа.
3. Убедитесь, что ключ выдан именно для этого экземпляра базы 1С.

---

### «Таймаут» / «Request Timeout»

**Симптом:** Запросы уходят, но завершаются по таймауту. В журнале — «Ошибка: время ожидания истекло».

**Решения:**
1. Увеличьте значение **Таймаут запроса** до `10–15` секунд в настройках интеграции.
2. Проверьте загруженность сервера NiFi у администратора инфраструктуры.
3. Убедитесь, что сеть между сервером 1С и NiFi не перегружена.

> Таймаут влияет только на ожидание ответа — если сообщение ушло, но ответ не получен, оно попадёт в очередь повторной отправки и будет доставлено при следующей попытке.

---

### Очередь растёт, сообщения не доставляются

**Симптом:** В журнале интеграции накапливаются записи со статусом «В очереди» или «Ошибка», количество увеличивается.

**Причины:**
- NiFi недоступен (плановые работы, сбой).
- Исчерпан лимит соединений на стороне NiFi.

**Действия:**
1. Уточните у администратора инфраструктуры статус NiFi.
2. Пока NiFi недоступен — сообщения накапливаются в очереди, потеря данных не происходит.
3. Когда NiFi восстановится — очередь разберётся автоматически в течение 1–2 минут.

> Максимальный размер очереди в базе данных — 10 000 сообщений. При переполнении новые сообщения не теряются, но система предупредит администратора.

---

### Подписки не срабатывают при проведении документов

**Симптом:** Заказы и отгрузки проводятся, но записи в журнале интеграции не появляются.

**Проверьте:**
1. Расширение активно (конфигуратор → Расширения конфигурации → галочка «Активно»).
2. База данных обновлена после активации расширения (шаг 2.4).
3. Пользователь, проводящий документы, не работает в эксклюзивном режиме.

**Если ничего не помогло:** перезапустите базу данных (попросите всех пользователей выйти, перезапустите сервер 1С).

---

### Входящие команды не обрабатываются

**Симптом:** Сервис рентабельности отправил команду (согласование, санкция), но в 1С изменений нет.

**Проверьте:**
1. Регламентное задание «Получение входящих команд» включено и выполняется (колонка «Последний запуск» обновляется).
2. В журнале интеграции нет ошибок для входящих сообщений.
3. Уточните у администратора сервиса рентабельности, что команда действительно была отправлена.

## 7. Откат и удаление расширения

### 7.1. Временное отключение расширения

Если необходимо приостановить работу интеграции без удаления расширения:

1. Откройте **Конфигуратор → Конфигурация → Расширения конфигурации**.
2. Снимите галочку **Активно** у расширения.
3. Обновите конфигурацию базы данных (**F7**).

После отключения:
- Подписки перестают срабатывать — события не отправляются.
- Регламентные задания остаются, но ничего не делают.
- Все настройки и накопленные данные очереди сохраняются.

Для повторного включения — верните галочку и обновите базу.

### 7.2. Полное удаление расширения

> **Внимание:** Удаление расширения удаляет все настройки и историю журнала интеграции. Перед удалением сохраните настройки (URL, API-ключ) и убедитесь, что все неотправленные сообщения из очереди доставлены.

**Порядок удаления:**

1. Отключите регламентные задания (снимите галочки «Разрешено»).
2. Дождитесь, пока очередь отправки опустеет (журнал интеграции → статус «В очереди» = 0).
3. Откройте **Конфигуратор → Конфигурация → Расширения конфигурации**.
4. Выберите расширение и нажмите кнопку **Удалить** (знак «–»).
5. Подтвердите удаление.
6. Обновите конфигурацию базы данных (**F7**).

**После удаления:**
- Все подписки на события документов прекращают работу.
- Константы подключения удаляются.
- Журнал интеграции очищается.
- Регламентные задания исчезают из списка.

## Приложение: топики событий

Для справки — перечень событий, которые расширение отправляет и получает.

### Исходящие события (9 топиков)

| Топик | Событие | Когда срабатывает |
|-------|---------|-----------------|
| `1c.order.created.v1` | Заказ создан | Запись нового документа «Заказ покупателя» |
| `1c.order.updated.v1` | Заказ обновлён | Изменение статуса или состава заказа |
| `1c.shipment.posted.v1` | Отгрузка проведена | Проведение «Реализации товаров и услуг» |
| `1c.shipment.returned.v1` | Возврат отгрузки | Проведение «Возврата товаров от покупателя» |
| `1c.price.npss-updated.v1` | Обновление НПСС | Пересчёт нетто-прайса стандартных затрат |
| `1c.price.purchase-changed.v1` | Изменение закупочной цены | Смена закупочной цены у поставщика |
| `1c.client.updated.v1` | Обновление клиента | Изменение карточки контрагента |
| `1c.ls.created.v1` | Создание листа согласования | Создание нового листа согласования |
| `1c.ls.plan-changed.v1` | Изменение плана ЛС | Пересчёт плановой рентабельности |

### Входящие команды (3 топика)

| Топик | Команда | Действие в 1С |
|-------|---------|--------------|
| `cmd.approval.result.v1` | Результат согласования | Обновить статус документа реализации |
| `cmd.sanction.applied.v1` | Применение санкции | Записать санкцию в карточку контрагента; при типе «block» — запретить создание новых заказов |
| `cmd.block.shipment.v1` | Блокировка отгрузки | Запретить проведение документа реализации |
