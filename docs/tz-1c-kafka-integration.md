# ТЗ для команды 1С: Kafka-интеграция с сервисом рентабельности

**Дата:** 2026-03-12
**Версия:** 1.0
**Платформа:** 1С:Управление торговлей 10.2 (платформа 8.3)
**Способ реализации:** Расширение конфигурации (.cfe)
**Транспорт:** Apache Kafka через NiFi (брокер сообщений)

---

## 1. Зачем это нужно

Сервис рентабельности отгрузок автоматически контролирует маржинальность каждой отгрузки. Для работы ему нужны данные из 1С:УТ в реальном времени:
- Когда создаётся заказ — проверить рентабельность
- Когда проводится отгрузка — зафиксировать факт
- Когда меняется НПСС — пересчитать маржу по всем активным листам согласования
- Когда обновляется клиент — актуализировать лимиты и стратегический статус

Обратно сервис передаёт в 1С **результаты согласования** и **санкции**.

---

## 2. Схема обмена

```
┌──────────────┐     Kafka (NiFi)      ┌────────────────────────┐
│              │  ──── 9 топиков ────►  │                        │
│  1С:УТ 10.2  │                        │  Сервис рентабельности │
│              │  ◄─── 3 топика ─────   │  (Go)                  │
└──────────────┘                        └────────────────────────┘
```

**1С отправляет** (9 топиков) — события при изменении данных.
**1С принимает** (3 топика) — команды от сервиса рентабельности.

## 3. Что 1С должна отправлять (9 исходящих событий)

### 3.1. Заказ создан — `1c.order.created.v1`

**Когда:** При записи нового документа "Заказ покупателя".

```json
{
  "event_id": "uuid-v4",
  "event_type": "order.created",
  "source": "1c-ut-10.2",
  "occurred_at": "2026-03-12T15:30:00+03:00",
  "external_id": "ORD-2026-00145",
  "local_estimate_id": "uuid листа согласования",
  "client_id": "uuid клиента",
  "client_name": "ООО Рога и Копыта",
  "manager_id": "uuid менеджера",
  "manager_name": "Иванов И.И.",
  "priority": "P2",
  "source_1c": "manual",
  "line_items": [
    {
      "product_id": "uuid товара",
      "product_name": "Автомат ВА47-29 1P 16A",
      "quantity": "100",
      "price": "245.50",
      "amount": "24550.00"
    }
  ],
  "total_amount": "24550.00"
}
```

**Правила:**
- `priority` = "P1" если сумма > 500 000 руб., иначе "P2"
- `source_1c` = "manual" (ручной ввод) или "edi" (электронный обмен)
- `local_estimate_id` — UUID листа согласования, к которому привязан заказ
- Все суммы и цены — строки с 2 десятичными знаками

### 3.2. Заказ обновлён — `1c.order.updated.v1`

**Когда:** При изменении статуса или состава заказа.

Формат аналогичен `order.created`, но `event_type` = `"order.updated"`, плюс поле:
```json
{
  "changes": ["status", "line_items", "total_amount"]
}
```

### 3.3. Отгрузка проведена — `1c.shipment.posted.v1`

**Когда:** При проведении документа "Реализация товаров и услуг".

```json
{
  "event_id": "uuid-v4",
  "event_type": "shipment.posted",
  "source": "1c-ut-10.2",
  "occurred_at": "2026-03-12T16:00:00+03:00",
  "external_id": "SHP-2026-00089",
  "order_id": "uuid заказа",
  "local_estimate_id": "uuid ЛС",
  "client_id": "uuid клиента",
  "manager_id": "uuid менеджера",
  "line_items": [
    {
      "product_id": "uuid товара",
      "product_name": "Автомат ВА47-29 1P 16A",
      "quantity": "100",
      "price": "245.50",
      "npss": "198.30",
      "amount": "24550.00"
    }
  ],
  "total_amount": "24550.00"
}
```

**Важно:** `npss` (НПСС) — нетто-прайс стандартных затрат, фиксируется в момент отгрузки.

### 3.4. Возврат отгрузки — `1c.shipment.returned.v1`

**Когда:** При проведении документа "Возврат товаров от покупателя".

Формат аналогичен `shipment.posted`, `event_type` = `"shipment.returned"`.

### 3.5. Обновление НПСС — `1c.price.npss-updated.v1`

**Когда:** При пересчёте НПСС (смена курса, закупочной цены, логистики).

```json
{
  "event_id": "uuid-v4",
  "event_type": "price.npss-updated",
  "source": "1c-ut-10.2",
  "occurred_at": "2026-03-12T09:00:00+03:00",
  "product_id": "uuid товара",
  "product_name": "Автомат ВА47-29 1P 16A",
  "old_npss": "198.30",
  "new_npss": "210.15",
  "purchase_price": "165.00",
  "logistics_cost": "25.15",
  "overhead_cost": "20.00",
  "method": "weighted_average",
  "trigger": "purchase_price"
}
```

**Trigger:** `planned` (плановый пересчёт), `exchange_rate` (курс), `purchase_price` (закупка).

### 3.7. Обновление клиента — `1c.client.updated.v1`

**Когда:** При изменении карточки контрагента.

```json

  "event_id": "uuid-v4",
  "event_type": "client.updated",
  "source": "1c-ut-10.2",
  "occurred_at": "2026-03-12T11:00:00+03:00",
  "external_id": "7701234567/770101001",
  "client_name": "ООО Рога и Копыта",
```

## 4. Что 1С должна принимать (3 входящих команды)

### 4.1. Результат согласования — `cmd.approval.result.v1`

```json
{
  "command_id": "uuid-v4",
  "command_type": "approval.result",
  "source": "profitability-service",
  "occurred_at": "2026-03-12T16:30:00+03:00",
  "shipment_id": "uuid отгрузки",
  "local_estimate_id": "uuid ЛС",
  "decision": "approved",
  "approver_id": "uuid одобрителя",
  "approver_name": "Петров П.П.",
  "approval_level": "dp",
  "comment": "Одобрено с учётом стратегического клиента"
}
```

**`decision`:** `approved` | `rejected` | `correction_required`

**Действие в 1С:** Обновить статус документа "Реализация товаров и услуг" или заблокировать проведение.

### 4.2. Применение санкции — `cmd.sanction.applied.v1`

```json
{
  "command_id": "uuid-v4",
  "command_type": "sanction.applied",
  "source": "profitability-service",
  "occurred_at": "2026-03-12T17:00:00+03:00",
  "client_id": "uuid клиента",
  "sanction_type": "warning",
  "reason": "Систематическое отклонение маржи ниже плана (3 отгрузки подряд)",
  "effective_date": "2026-03-13",
  "expires_at": "2026-04-13"
}
```

**`sanction_type`:** `warning` | `limit_reduction` | `block`

**Действие в 1С:** Записать информацию в карточку контрагента, при `block` — запретить создание новых заказов.

### 4.3. Блокировка отгрузки — `cmd.block.shipment.v1`

```json
{
  "command_id": "uuid-v4",
  "command_type": "block.shipment",
  "source": "profitability-service",
  "occurred_at": "2026-03-12T17:30:00+03:00",
  "shipment_id": "uuid отгрузки",
  "reason": "Маржинальность ниже допустимого порога (3.2% при минимуме 5%)",
  "blocked_by": "system"
}
```

**Действие в 1С:** Запретить проведение документа реализации.

## 5. Технические требования

### 5.1. Транспорт: Kafka через NiFi

- **Брокер:** Apache Kafka (адрес предоставляется при настройке)
- **Промежуточный слой:** Apache NiFi (маршрутизация, трансформация)
- **Формат:** JSON (UTF-8)
- **Кодировка дат:** ISO 8601 с часовым поясом МСК (+03:00)
- **Все суммы:** строки с 2 десятичными знаками ("245.50", не 245.5)
- **UUID:** формат v4 (xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx)

### 5.2. Обязательные Kafka-заголовки (headers)

Каждое сообщение ДОЛЖНО содержать два заголовка:

| Заголовок | Значение | Пример |
|-----------|----------|--------|
| `message_id` | UUID-v4, уникальный для каждого сообщения | `a1b2c3d4-e5f6-4a7b-8c9d-0e1f2a3b4c5d` |
| `correlation_id` | UUID-v4 бизнес-операции (одинаковый для связанных событий) | `f1e2d3c4-b5a6-4789-0123-456789abcdef` |

**`message_id`** используется для дедупликации: повторная отправка того же `message_id` игнорируется (TTL 7 дней).
**`correlation_id`** используется для сквозной трассировки: все события одного бизнес-процесса имеют один `correlation_id`.

Если 1С использует HTTP POST в NiFi — заголовки передаются как HTTP-заголовки `X-Message-Id` и `X-Correlation-Id`. NiFi маппит их в Kafka headers.

### 5.3. Гарантии доставки

- **At-least-once:** каждое сообщение гарантированно доставляется минимум 1 раз
- **Идемпотентность:** сервис рентабельности дедуплицирует по `event_id` (TTL 7 дней)
- **Порядок:** сообщения по одному `client_id` или `local_estimate_id` приходят в порядке (один partition)

### 5.4. Требования к расширению 1С

- Реализовать как расширение (.cfe), НЕ менять типовую конфигурацию
- Подписки на события: ПриЗаписи/ПослеЗаписи соответствующих документов
- Формирование JSON и отправка в NiFi через HTTP-сервис (NiFi слушает HTTP)
- Приём команд: регламентное задание, опрашивающее NiFi/Kafka REST Proxy

### 5.5. Что НЕ нужно делать в 1С

- НЕ реализовывать логику расчёта рентабельности (это делает Go-сервис)
- НЕ реализовывать логику согласования (это делает Go-сервис)
- НЕ хранить историю согласований (хранит Go-сервис)
- Только: отправлять события + принимать команды

## 6. Тестирование

- В Go-сервисе есть **1С-симулятор** (`cmd/1c-simulator/main.go`) — генерирует тестовые события
- Контрактные тесты проверяют формат JSON (уже написаны)
- При разработке можно проверять формат через симулятор до подключения к реальному Kafka
