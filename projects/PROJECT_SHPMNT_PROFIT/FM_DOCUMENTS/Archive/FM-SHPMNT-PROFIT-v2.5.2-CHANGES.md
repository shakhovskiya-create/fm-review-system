## ДЕТАЛЬНЫЕ ПРАВКИ

### 1. [CRIT-001] п. 3.5.5 — Дефолт-решения при отсутствии реакции

**БЫЛО:**
```
ГД 24ч → автоотказ (исключение: при рентабельности заказа ≥0% — автосогласование)
```

**СТАЛО:**
```
ГД 24ч → автоотказ (исключение: при (Накопленная + Заказ) ≥ плановой
рентабельности ЛС — автосогласование).

Обоснование: для убыточных ЛС (план < 0%) условие "рент. заказа ≥ 0%"
приводит к автоотказу даже при улучшении показателей. Новая формулировка
корректно обрабатывает все случаи.
```

---

### 2. [CRIT-002] п. 3.10 — Точка невозврата

**БЫЛО:**
```
После статуса «Передан на склад» согласование НЕ аннулируется
```

**СТАЛО:**
```
Точка невозврата — передача на склад:

При передаче задания в WMS устанавливается реквизит «ПереданНаСклад» (Булево)
и фиксируется «ДатаПередачиНаСклад». После установки флага:
• статус документа остаётся «Согласован»;
• согласование НЕ аннулируется (даже при истечении 5 р.д. или изменении
  рентабельности по другим заказам);
• отмена возможна только через специалиста документооборота склада
  после удаления задания из WMS.

Примечание: отдельный статус "Передан на склад" не вводится для упрощения
схемы переходов.
```

### 3. [CRIT-003] п. 3.4 — Учёт параллельных заказов

**БЫЛО:**
```
При расчёте показателей учитываются:
в (Накопл. + Заказ) — только фактически отгруженные РТУ + текущий Заказ;
```

**СТАЛО:**
```
При расчёте показателей учитываются:
в (Накопл. + Заказ) — фактически отгруженные РТУ + Заказы в статусе
«На согласовании» (как условно согласованные) + текущий Заказ.

Это означает: если по ЛС уже есть заказ на согласовании, новый заказ
учитывает его влияние на рентабельность. Это исключает ситуацию, когда
несколько параллельных заказов проходят проверку независимо, а в сумме
дают неприемлемый результат.

Пример:
• Заказ-1 (откл. 8 п.п.) отправлен на согласование РБЮ
• Менеджер создаёт Заказ-2
• Система рассчитывает (Накопл. + Заказ-1 + Заказ-2)
• Если суммарное отклонение превышает порог — Заказ-2 также требует согласования
```

### 4. [CRIT-004] SHPMNT-058 — Привязка возвратов

**БЫЛО:**
```
Возвраты 30д — документы «Возврат товаров» с датой возврата ≤ 30 дней от даты РТУ.
```

**СТАЛО:**
```
Возвраты 60д — документы «Возврат товаров» с датой возврата ≤ 60 дней
от даты РТУ, на основании которого оформлен возврат (по реквизиту
«Документ-основание» в документе возврата).

При множественных отгрузках одной номенклатуры привязка определяется
по документу-основанию возврата. Если документ-основание не указан —
по методу FIFO (к самому раннему РТУ с данной номенклатурой).

Обоснование увеличения срока: 30 дней недостаточно для выявления
манипуляций «отгрузить для % выкупа → вернуть».
```

**Также изменить в п. 3.16.8:**
- Все упоминания "Возвраты 30д" → "Возвраты 60д"
- В формуле: (Отгружено − Возвраты 60д)

### 5. [HIGH-001] п. 3.7.4 — Владелец EDI-заказа

**ДОБАВИТЬ после "владелец сделки принимает решение":**

```
Определение владельца EDI-заказа:
• Если EDI-заказ создан вручную менеджером — владелец = инициатор;
• Если EDI-заказ поступил автоматически — владелец = менеджер,
  закреплённый за клиентом в справочнике «Ответственные менеджеры
  по контрагентам»;
• При отсутствии закреплённого менеджера — владелец = дежурный менеджер
  БЮ (по графику дежурств, ведётся в справочнике «График дежурств БЮ»).

Владелец сделки несёт ответственность за коммуникацию с клиентом
и принятие решений в рамках 4-часового окна.
```

### 6. [HIGH-002] п. 3.8 и п. 3.12 — Унификация пересогласования

**В п. 3.8 ИЗМЕНИТЬ:**
```
БЫЛО:
если новая рентабельность требует более высокого уровня согласования →
требуется пересогласование ЛС

СТАЛО:
при снижении плановой рентабельности ЛС требуется согласование
по матрице п. 3.14.3 (снижение 0–3,00 п.п. → РБЮ; 3,01–10,00 п.п. → ДП;
более 10,00 п.п. → ГД).
```

**В п. 3.12 ИЗМЕНИТЬ:**
```
БЫЛО:
если новая плановая рентабельность ниже исходной — требуется повторное
согласование ЛС

СТАЛО:
при снижении плановой рентабельности ЛС применяется матрица согласования
из п. 3.14.3. Повышение плановой рентабельности согласования не требует.
```

### 7. [HIGH-003] п. 3.8 — Автопродление и лимит

**ДОБАВИТЬ после "суммарный срок продлений учитывается":**

```
Автопродление по п. 3.16.6 (при позднем поступлении товара):
• НЕ входит в лимит 2 ручных продлений;
• НЕ влияет на пересчёт НПСС;
• учитывается отдельно как техническое продление (реквизит
  «КоличествоАвтопродлений» в документе ЛС).

Обоснование: автопродление — вынужденная мера из-за дефицита компании.
Клиент не должен нести ответственность за проблемы поставщика.
```

### 10. [MED-003] п. 3.2а — НПСС = NULL

**ДОБАВИТЬ после формулы рентабельности:**

```
Обязательное условие расчёта:

Если НПСС = 0, NULL или не заполнена для позиции Заказа:
• создание Заказа блокируется;

Это предотвращает деление на ноль и некорректные расчёты рентабельности.
```

### 17. [NF-004] п. 3.11 — Гонка согласование vs истечение ЛС

**ДОБАВИТЬ после "Заказы в статусе «На согласовании» — автоматически аннулируются":**

```
Защита от гонки "согласование vs истечение":

Если ЛС истекает в момент, когда согласующий принимает решение:
• система проверяет статус ЛС непосредственно перед записью решения;
• если ЛС уже истекла — решение не фиксируется, задача закрывается
  с результатом «Отменено (истечение ЛС)»;
• согласующий получает уведомление: «ЛС истекла. Согласование невозможно»;
• Заказ переходит в статус «Согл. истекло».

Это исключает фиксацию согласования по истекшей ЛС.
```
