### #1: PLAT-001 — Отсутствие указания платформы в ФМ

**Источник:** Аудит v1.0.5
**Severity:** MEDIUM
**Тип:** C (пробел, принято)
**Статус:** Принято -- требуется правка ФМ

**Суть замечания:** ФМ не содержит явного указания на платформу 1С:УТ 10.2 и ограничения обычных форм. Разработчик может ошибочно предположить работу на управляемых формах.

**Анализ:** Замечание корректно. ФМ описывает бизнес-процесс, но не фиксирует платформенный контекст, хотя ТЗ v3.0 и Архитектура v3.0 полностью переработаны с учетом обычных форм. Для целостности документации в разделе 1 ФМ необходимо указание платформы.

**Решение (без замедления процесса):**

Добавить в раздел 1 (Общие сведения) абзац:

> Система реализуется как расширение (.cfe) для 1С:Управление Торговлей, редакция 10.2, на платформе 1С:Предприятие 8.3 (режим обычного приложения). Все формы -- обычные (толстый клиент). Детали платформенных ограничений и их влияние на реализацию описаны в ТЗ (раздел 1.7) и Архитектуре (раздел 0).

**Правка для ФМ:**

| Параметр | Значение |
|----------|----------|
| Где | Раздел 1 (Общие сведения), после описания назначения |
| Было | Отсутствует указание платформы |
| Станет | Абзац с указанием платформы, режима и ссылкой на ТЗ/Архитектуру |
| Влияние на скорость | +0 мин. Информационный текст, не добавляет шагов. |

### #2: PLAT-002 — Обратный отсчет SLA на обычных формах

**Источник:** Аудит v1.0.5
**Severity:** MEDIUM
**Тип:** A (учтено в ТЗ v3.0)
**Статус:** Отклонено -- уже покрыто в ТЗ и Архитектуре

**Суть замечания:** Обратный отсчет SLA предполагает динамическое обновление UI. Необходимо убедиться, что ТЗ описывает реализацию через таймер обычной формы.

**Анализ:** Замечание закрыто. ТЗ v3.0 и Архитектура v3.0 полностью переработаны с учетом обычных форм:

- **Архитектура, раздел 0.4:** описан механизм ПодключитьОбработчикОжидания для обновления индикаторов на обычных формах. Надписи с программной подсветкой цветом (ЦветТекста), обновляемые таймером.
- **ТЗ, раздел 1.7.1:** описан механизм уведомлений через модальные диалоги (Предупреждение/Вопрос) + надписи на формах.
- **Feasibility review:** PLAT-002 = feasible, effort medium. Цитата: "Индикаторы нагрузки на обычной форме: надписи с программной подсветкой цветом через ЦветТекста. Обновление через ПодключитьОбработчикОжидания."
- **Тест-кейс TC-075:** явно проверяет таймер ПодключитьОбработчикОжидания.

**Вывод:** ФМ описывает бизнес-функцию (обратный отсчет SLA), а техническая реализация корректно описана в ТЗ и Архитектуре. Правка ФМ не требуется -- разделение зон ответственности документов соблюдено.

### #5: LOW-003 — Непоследовательные кавычки

**Источник:** Аудит v1.0.5
**Severity:** LOW
**Тип:** D (бэклог)
**Статус:** Отложено на следующую итерацию

**Суть замечания:** В документе используются разные типы кавычек: елочки и лапки для статусов.

**Анализ:** Замечание косметическое. Унификация кавычек по всему документу (50+ страниц) -- трудоемкая правка с риском нарушить форматирование в Confluence (XHTML). При этом влияние на понятность документа минимальное -- все статусы однозначно читаемы.

**Решение:** Зафиксировано в бэклоге. Унификация кавычек будет выполнена при следующем крупном обновлении ФМ, когда документ будет пересмотрен целиком.

### #8: Риск -- регрессионное тестирование 2-3 дня вручную

**Источник:** Тестирование v1.0.5, риски
**Тип:** D (бэклог пилота)
**Статус:** Зафиксировано

**Суть:** 80 тест-кейсов выполняются вручную. При каждом обновлении платформы 8.3 -- полный прогон (2-3 рабочих дня).

**Анализ:** Это объективное ограничение платформы 1С:УТ 10.2 (обычные формы). Средства автоматизации UI-тестирования (Vanessa Automation, TestClient) работают только с управляемыми формами.

**Решение в ФМ не требуется.** ФМ описывает бизнес-процесс, а не стратегию тестирования. Риск зафиксирован в ТЗ v3.0, раздел 9.2 и учтен в оценке трудоемкости (35% на тестирование вместо стандартных 25-30% при наличии автотестов).

### #9: Риск -- автоматизация UI невозможна

**Источник:** Тестирование v1.0.5, риски
**Тип:** D (бэклог пилота)
**Статус:** Зафиксировано

**Суть:** Vanessa Automation и TestClient не поддерживают обычные формы УТ 10.2.

**Анализ:** Дублирует риск #8. Принятое ограничение платформы. Серверная бизнес-логика (расчеты рентабельности, движения регистров, интеграции) тестируется через xUnitFor1C (12 из 80 тест-кейсов). Это покрывает критичную часть -- правильность вычислений и бизнес-правил.

**Решение:** ТЗ v3.0, раздел 9.1 описывает способ обхода: "Ручное тестирование (72 тест-кейса). Серверная логика тестируется через xUnitFor1C." ФМ не требует изменений.

### #12: Оценка трудоемкости: 3 675 ч (+33% vs базовая)

**Источник:** ТЗ v3.0
**Тип:** B (осознанный выбор)
**Статус:** Подтверждено -- увеличение обоснованно

**Суть:** Оценка трудоемкости увеличена с 2 758 ч (v2.0) до 3 675 ч (v3.0), рост +917 ч (+33%).

**Анализ увеличения:**

| Фактор | Описание | Коэффициент |
|--------|----------|-------------|
| Разработка форм | Программное создание элементов в ПриОткрытии, нет визуального редактора для расширяемых форм, каждая кнопка -- отдельная процедура | +40% |
| Отладка | Единый контекст, менее информативные ошибки платформы | +30% |
| Тестирование | Автоматизация UI невозможна, только ручное тестирование | +35% |
| Документирование | Нестандартные решения, описание обходов | +20% |
| **Средневзвешенный** | | **+35%** |

**Детализация P1/P2:**

| Фаза | Часы | Команда | Срок |
|------|------|---------|------|
| P1 (MVP) | 3 037 ч | 3 разработчика + 1 аналитик + 1 тестировщик | ~6 мес. |
| P2 (Этап 2) | 638 ч | 2 разработчика + 1 тестировщик | ~2 мес. |
| **Итого** | **3 675 ч** | | **~8 мес.** |

**Позиция:** Увеличение обоснованно. Коэффициент +35% на ограничения обычных форм подтвержден:
1. Отсутствие визуального редактора для расширяемых форм -- все элементы создаются программно.
2. Невозможность автотестирования UI -- 72 тест-кейса вручную при каждом обновлении.
3. Прекращение поддержки вендором (с 01.04.2024) -- обход ошибок типового кода силами команды.

**Альтернатива (без увеличения бюджета):** Миграция на УТ 11.x + управляемые формы. Оценка миграции: +400-600 ч (отдельный проект, 12-18 мес.). Это не ускорит текущий проект, но даст экономию на последующей поддержке.
