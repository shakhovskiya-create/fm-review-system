# Регистр сведений: ekf_ОбработанныеКоманды

**Назначение:** Дедупликация входящих команд от profitability-service. При повторной отправке команды с тем же `command_id` HTTP-сервис возвращает 200 OK без повторной обработки.

**Задача:** EKFLAB-519
**Автор:** Шаховский А.С.

## Структура

| Поле | Роль | Тип | Индекс | Описание |
|------|------|-----|--------|----------|
| ИдентификаторКоманды | Измерение | Строка(36) | Да | command_id (UUID v4) |
| ДатаОбработки | Ресурс | Дата | Нет | Время обработки команды |
| ТипКоманды | Ресурс | Строка(30) | Нет | `approval` / `sanction` / `block` |
| Результат | Ресурс | Строка(10) | Нет | `ok` / `error` |

## Жизненный цикл

```
1. HTTP POST /profitability/api/v1/commands -> парсинг JSON
2. Проверка: command_id уже в регистре? -> Да: вернуть 200 (idempotent)
3. Обработка команды бизнес-логикой
4. Запись command_id в регистр (ok / error)
5. Очистка: записи старше 30 дней удаляются фоновым заданием
```

## TTL

Записи хранятся 30 дней. Фоновое задание `ekf_ОчисткаСтарыхЗаписей` (ежедневно 03:00) вызывает `ekf_ИнтеграцияФоновые.ОчиститьСтарыеКоманды()`.

## Типы команд (3)

| command_type | Топик Kafka | Действие в 1С |
|-------------|-------------|---------------|
| `approval.result` | `cmd.approval.result.v1` | Обновить статус реализации |
| `sanction.applied` | `cmd.sanction.applied.v1` | Записать санкцию в контрагента |
| `block.shipment` | `cmd.block.shipment.v1` | Заблокировать/распровести реализацию |
