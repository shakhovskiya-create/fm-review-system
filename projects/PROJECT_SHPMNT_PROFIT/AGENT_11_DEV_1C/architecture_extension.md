## 3. Маппинг событий на объекты 1С

| Событие | Топик Kafka | Источник в 1С | Подписка | Триггер |
|---------|-------------|---------------|----------|---------|
| Заказ создан | `1c.order.created.v1` | Документ.ЗаказПокупателя | ekf_ПриЗаписиЗаказаПокупателя | ПриЗаписи (новый) |
| Заказ обновлен | `1c.order.updated.v1` | Документ.ЗаказПокупателя | ekf_ПриЗаписиЗаказаПокупателя | ПриЗаписи (существующий) |
| Отгрузка | `1c.shipment.posted.v1` | Документ.РеализацияТоваровИУслуг | ekf_ПриПроведенииРеализации | ОбработкаПроведения |

| НПСС обновлена | `1c.price.npss-updated.v1` | Документ.УстановкаЦенНоменклатуры | ekf_ПриЗаписиУстановкаЦен | ПриЗаписи |
| Закупочная цена | `1c.price.purchase-changed.v1` | Документ.УстановкаЦенНоменклатуры | ekf_ПриЗаписиУстановкаЦен | ПриЗаписи |

## 4. Решения по retry и обработке ошибок

### 4.1. Retry-стратегия (исходящие события)

| Попытка | Задержка | Комментарий |
|---------|----------|-------------|
| 1 | 30 сек | Быстрый retry (кратковременный сбой) |
| 2 | 1 мин | |
| 3 | 2 мин | |
| 4 | 5 мин | |
| 5 | 10 мин | |
| 6-10 | 15 мин | Средние сбои |
| 11-20 | 30 мин | |
| 21-50 | 1 час | Длительная недоступность |
| 51-100 | 2 часа | |
| >100 | expired | Сообщение помечается expired, алерт в журнал |

**Общее время retry:** ~6 дней до перехода в expired.

### 4.2. Классификация HTTP-ответов

| HTTP код | Действие | Повторять? |
|----------|----------|------------|
| 200/202 | Успех | Нет |
| 409 | Дубликат (уже получено) | Нет |
| 400 | Невалидный payload | Нет (permanent failure, лог ERROR) |
| 401 | Неверный API key | Да (может быть ротация ключа) |
| 429 | Rate limit | Да (с увеличенной задержкой) |
| 500/502/503 | Ошибка сервера | Да |
| Таймаут/сеть | Сетевая ошибка | Да |

### 4.3. Гарантии доставки

- **At-least-once**: каждое событие гарантированно доставлено минимум 1 раз
- **Идемпотентность**: profitability-service дедуплицирует по `event_id` (TTL 7 дней)
- **Порядок**: события одного документа отправляются последовательно (одна подписка = один вызов)
- **Kill switch**: константа `ekf_ИнтеграцияАктивна` = Ложь мгновенно останавливает все отправки

### 4.4. Обработка входящих команд

- **Идемпотентность**: проверка `command_id` в регистре `ekf_ОбработанныеКоманды`
- **Аутентификация**: проверка `X-Api-Key` в заголовке запроса
- **Транзакционность**: каждая команда обрабатывается в отдельной транзакции с блокировкой данных
- **Журналирование**: все входящие команды логируются в журнал регистрации

## 5. Формат JSON-сообщений

### 5.1. Конверт исходящего события

Каждое сообщение отправляется как HTTP POST body:

```json
{
  "event_id": "a1b2c3d4-e5f6-4a7b-8c9d-0e1f2a3b4c5d",
  "event_type": "order.created",
  "source": "1c-ut-10.2",
  "occurred_at": "2026-03-13T15:30:00+03:00",
  ...поля по схеме топика...
}
```

HTTP-заголовки:
- `Content-Type: application/json`
- `X-Api-Key: <значение из ekf_ИнтеграцияNiFiAPIКлюч>`
- `X-Message-Id: <UUID v4>` (уникальный для каждого сообщения)
- `X-Correlation-Id: <UUID v4>` (единый для связанных событий бизнес-процесса)

### 5.2. Правила формирования данных

- **Суммы и цены**: строки с 2 десятичными знаками ("245.50", НЕ 245.5 и НЕ копейки)
- **UUID**: формат v4 (xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx)
- **Даты**: ISO 8601 с часовым поясом МСК (+03:00)
- **Количество**: строка ("100", НЕ число)
- **allowed_deviation**: строка ("7", НЕ число)
- **Пустые значения**: пустая строка "", НЕ null
