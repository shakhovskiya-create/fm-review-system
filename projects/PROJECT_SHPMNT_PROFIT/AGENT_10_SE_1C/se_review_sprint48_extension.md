## СЕКЦИЯ 1: Architecture Review

### [HIGH-001] Регистр ekf_ОчередьОтправкиСобытий: строковый статус вместо перечисления

**Файл:** `register_event_queue.md`, `module_kafka_integration.bsl` (строка 547), `scheduled_job_retry.bsl` (строки 43, 66, 79, 86, 121)

**Описание:** Поле `Статус` имеет тип `Строка(20)` со значениями "pending" / "sending" / "expired". Строковые статусы -- антипаттерн в 1С: нет контроля допустимых значений, возможны опечатки ("Pending", "PENDING"), усложненный поиск ошибок.

**Почему важно:** При 50-70 одновременных пользователях и регламентном задании каждые 60 секунд любая опечатка в статусе приведет к "зависанию" записи в очереди без возможности обнаружения стандартными средствами.

**Варианты решения:**

| Вариант | Трудоемкость | Риск | Влияние | Поддержка |
|---------|-------------|------|---------|-----------|
| А: Перечисление ekf_СтатусОчередиОтправки (3 значения) | Низкая (1ч) | Низкий | Высокое | Низкая |
| Б: Оставить строку + валидация в ПоместитьВОчередь | Низкая (0.5ч) | Средний | Среднее | Средняя |
| В: Ничего не менять | -- | -- | -- | Накопление долга |

**Рекомендация:** Вариант А. Перечисление гарантирует type safety на уровне платформы. Это 18 объектов метаданных -- перечисление станет 19-м, трудоемкость минимальная.

### [HIGH-002] Чтение констант на каждый вызов подписки -- нагрузка на кеш

**Файл:** `module_kafka_integration.bsl` (строки 27, 48), `subscriptions_order.bsl` (строка 28), `subscriptions_shipment.bsl` (строка 27), `subscriptions_return.bsl` (строка 27), `subscriptions_prices.bsl` (строка 28), `subscriptions_client.bsl` (строка 26)

**Описание:** Каждая подписка и каждый вызов `ОтправитьСобытие` читает константу `ekf_ИнтеграцияАктивна.Получить()`. При записи заказа это происходит дважды (подписка + ОтправитьСобытие). Аналогично в `ОтправитьHTTPВNiFi` читаются 3 константы (URL, APIKey, Таймаут). При 50-70 пользователях и высокой частоте записей -- десятки обращений к константам в секунду.

**Почему важно:** Хотя платформа 8.3 кеширует константы в сеансе, двойное чтение kill switch (подписка + модуль) -- избыточно. При отключенной интеграции каждая запись документа все равно выполняет 2 чтения из кеша.

**Варианты решения:**

| Вариант | Трудоемкость | Риск | Влияние | Поддержка |
|---------|-------------|------|---------|-----------|
| А: Убрать проверку kill switch из подписок, оставить только в ОтправитьСобытие | Низкая (0.5ч) | Низкий | Среднее | Низкая |
| Б: Кеширующая обертка через общий модуль с ПовторногоИспользования = ВызовахСеанса | Средняя (2ч) | Низкий | Высокое (URL, APIKey, Таймаут) | Низкая |
| В: Ничего не менять | -- | -- | -- | Двойное чтение |

**Рекомендация:** Вариант Б. Создать кеширующий модуль `ekf_ИнтеграцияНастройкиПовторноИспользуемые` (ПовторногоИспользования = ВызовахСеанса) с функциями: `ИнтеграцияАктивна()`, `АдресNiFi()`, `APIКлюч()`, `ТаймаутHTTP()`. Тогда все 4 константы читаются из БД только 1 раз за сеанс. Важно: при изменении констант администратором нужен механизм сброса кеша (например, через `ОбновитьПовторноИспользуемыеЗначения()`), но это стандартная практика.

### [HIGH-003] HTTP-вызов в контексте транзакции подписки ОбработкаПроведения

**Файл:** `subscriptions_shipment.bsl` (строка 32-35), `subscriptions_return.bsl` (строка 32-35)

**Описание:** Подписки `ekf_ПриПроведенииРеализации` и `ekf_ПриПроведенииВозврата` привязаны к событию `ОбработкаПроведения`. Это означает, что код подписки выполняется ВНУТРИ транзакции проведения документа. Вызов HTTP из подписки (через `ОтправитьСобытие` -> `ОтправитьHTTPВNiFi`) выполняется внутри этой транзакции. Таймаут HTTP = 5 секунд. Если NiFi недоступен -- транзакция проведения удерживается 5 секунд, блокируя другие операции с документом.

**Почему важно:** При таймауте HTTP = 5 секунд + retry в catch (помещение в очередь = запись в РС) -- транзакция проведения может растянуться до 5+ секунд. При 70 пользователях, проводящих реализации, это создает очереди ожидания и блокировки таблиц регистров накопления.

**Варианты решения:**

| Вариант | Трудоемкость | Риск | Влияние | Поддержка |
|---------|-------------|------|---------|-----------|
| А: Всегда записывать в очередь из подписок ОбработкаПроведения, HTTP-отправку делать только в фоновом задании | Средняя (3ч) | Низкий | Высокое | Низкая |
| Б: Снизить таймаут до 1-2 секунд для подписок проведения | Низкая (0.5ч) | Средний | Среднее | Средняя |
| В: Заменить ОбработкаПроведения на ПослеЗаписи для подписок shipment/return | Низкая (0.5ч) | Средний | Среднее | Средняя |

**Рекомендация:** Вариант А -- самый надежный. Подписка проведения всегда записывает в очередь (мгновенно, ~1мс), фоновое задание обрабатывает очередь каждые 60 секунд. Это гарантирует нулевое влияние интеграции на время проведения документа. Дополнительное преимущество: единый путь отправки (всегда через очередь) упрощает отладку и мониторинг.

### [HIGH-004] Отсутствие функциональной опции для включения/отключения расширения

**Файл:** `architecture_extension.md`

**Описание:** Расширение использует константу `ekf_ИнтеграцияАктивна` как kill switch, что корректно. Однако отсутствует функциональная опция, привязанная к этой константе. Функциональная опция в 1С позволяет скрывать элементы интерфейса (колонки, кнопки) и отключать видимость объектов расширения в менюв.

**Почему важно:** Без функциональной опции при отключенной интеграции пользователи по-прежнему видят все элементы расширения в интерфейсе (если таковые появятся в будущем), что вводит в заблуждение. Также функциональная опция -- стандартный механизм 1С для управления подсистемами.

**Варианты решения:**

| Вариант | Трудоемкость | Риск | Влияние | Поддержка |

| А: Создать ФункциональнуюОпцию ekf_ИнтеграцияРентабельности, привязать к константе | Низкая (1ч) | Низкий | Среднее | Низкая |
| Б: Ничего не менять (интеграция серверная, UI элементов нет) | -- | -- | -- | -- |

**Рекомендация:** Вариант А. Стандарт БСП требует функциональную опцию для подключаемого функционала. Даже если сейчас UI нет, опция подготовит расширение к будущему развитию.

### [MED-001] Строковые литералы на английском без предопределенных значений

**Файл:** `module_kafka_integration.bsl` (строки 547, 564-572), `http_service_commands.bsl` (строки 71-80)

**Описание:** Типы событий ("order.created", "shipment.posted") и команд ("approval.result", "sanction.applied") заданы строковыми литералами в коде. При добавлении нового типа события или опечатке в строке -- ошибку обнаружить сложно. Функция `ОпределитьТопик` дублирует эту информацию.

**Почему важно:** При 9 типах событий + 3 типах команд = 12 строковых литералов, разбросанных по 7 файлам. Рефакторинг или добавление нового типа потребует изменения в нескольких местах.

**Рекомендация:** Вынести типы событий и топики в отдельную функцию-справочник (или экспортные функции модуля). Например:
```bsl
Функция ТипСобытияЗаказСоздан() Экспорт
    Возврат "order.created";
КонецФункции
```
Это централизует управление строковыми литералами.

### [MED-002] ПолучитьНПСС: поиск типа цен по наименованию -- хрупкая привязка

**Файл:** `module_kafka_integration.bsl` (строка 740), `subscriptions_prices.bsl` (строка 171)

### [MED-003] Глотание исключений без логирования

**Файл:** `module_kafka_integration.bsl` (строки 742-743, 793-796, 816-818, 840-842, 860-864), `subscriptions_prices.bsl` (строки 91-93, 131-134, 206-208)

**Описание:** Множество блоков `Попытка / Исключение` с пустым телом обработки исключения. Примеры:
- `ОпределитьИсточникЗаказа`: `Исключение // Реквизит ВидЗаказа может отсутствовать КонецПопытки`
- `ПолучитьИдЛС`: `Исключение // Реквизит может отсутствовать КонецПопытки`
- `ОбработатьИзменениеНПСС`: `Исключение // Компоненты недоступны КонецПопытки`

**Почему важно:** Пустые обработчики исключений маскируют реальные ошибки. Если реквизит `ЛокальнаяСмета` не просто "отсутствует", а возникла ошибка доступа к объекту (например, объект удален) -- разработчик не узнает об этом.

**Рекомендация:** Добавить журналирование на уровне `Примечание` (не `Предупреждение` -- чтобы не засорять журнал):
```bsl
Исключение
    Если Константы.ekf_ИнтеграцияЛогированиеПодробное.Получить() Тогда
        ЗаписьЖурналаРегистрации("ekf_Интеграция.ОтладкаРеквизитов",
            УровеньЖурналаРегистрации.Примечание, , ,
            ПодробноеПредставлениеОшибки(ИнформацияОбОшибке()));
    КонецЕсли;
КонецПопытки;
```

### [MED-004] Нет проверки максимальной длины строки при формировании JSON

**Файл:** `module_kafka_integration.bsl` (функции СформироватьДанные*)

**Описание:** Функции формирования данных не проверяют длину строковых значений. Например, `Строка(ДокументОбъект.Контрагент)` может вернуть наименование длиной более 200 символов. Аналогично `Строка(СтрокаТовара.Номенклатура)`. Хотя JSON не ограничивает длину строк, Go-сервис может иметь ограничения на уровне базы данных (VARCHAR).

**Почему важно:** Потенциально длинные строки в JSON могут привести к ошибкам на стороне Go-сервиса при записи в PostgreSQL.

**Рекомендация:** Добавить `Лев(..., 200)` для строковых полей product_name, client_name, manager_name -- либо согласовать максимальные длины с Go-стороной. Низкая вероятность проблемы, но лучше предусмотреть.

### [MED-005] Запрос в подписке ПриЗаписи (ОпределитьИзмененияЗаказа)

**Файл:** `subscriptions_order.bsl` (строки 63-124)

**Описание:** Функция `ОпределитьИзмененияЗаказа` выполняет запрос к базе данных (`Документ.ЗаказПокупателя`) для сравнения текущего и сохраненного состояния документа. Запрос выполняется в контексте подписки `ПриЗаписи`, которая находится внутри транзакции записи. Это означает, что запрос прочитает уже измененные данные (если запись выполнена, но транзакция еще не зафиксирована -- запрос в той же транзакции увидит новые данные).

**Почему важно:** Запрос `Документ.ЗаказПокупателя ГДЕ Ссылка = &Ссылка` в контексте ПриЗаписи видит СТАРЫЕ данные (до текущей записи), но только если используется версионирование данных (MVCC). В 1С при записи в ПриЗаписи данные ещё не зафиксированы в БД, и запрос МОЖЕТ вернуть старые значения -- это зависит от версии платформы и СУБД. Поведение ненадежное.

**Рекомендация:** Использовать `ДополнительныеСвойства` документа для передачи "старых" значений. В обработчике `ПередЗаписью` (другая подписка расширения) сохранить снимок значений: `Источник.ДополнительныеСвойства.Вставить("ekf_СтарыеДанные", Структура)`. В `ПриЗаписи` сравнивать с этим снимком. Это надежный паттерн, гарантирующий корректное определение изменений.

### [MED-007] Статус "sending" не обрабатывается при рестарте сервера

**Файл:** `scheduled_job_retry.bsl` (строка 43)

**Описание:** Запрос очереди выбирает только записи со статусом "pending". Если сервер 1С перезапустился во время обработки записи (статус = "sending"), такая запись "зависнет" навсегда -- не будет обработана повторно и не будет очищена (очистка удаляет только "expired").

**Почему важно:** При перезапуске сервера 1С (плановом или аварийном) все записи со статусом "sending" становятся "потерянными". Со временем их количество может накапливаться.

**Рекомендация:** В начале `ОбработатьОчередьОтправки` добавить сброс "зависших" записей:
```bsl
// Сбросить записи в статусе "sending" старше 5 минут (зависшие после рестарта)
СброситьЗависшиеЗаписи();
```
Логика: если запись в статусе "sending" более 5 минут -- вернуть в "pending".

### [MED-008] HTTP-сервис: проверка HTTP-метода избыточна

**Файл:** `http_service_commands.bsl` (строки 34-37)

**Описание:** Проверка `Если ВРег(Запрос.HTTPМетод) <> "POST"` избыточна, потому что HTTP-сервис в 1С настраивается на уровне шаблона URL с указанием допустимых методов. Если шаблон `/commands` настроен на POST -- платформа вернет 405 до вызова обработчика.

**Почему важно:** Не является ошибкой, но загромождает код. Удаление этой проверки упростит обработчик.

**Рекомендация:** Удалить проверку HTTP-метода. Платформа 1С автоматически возвращает 405 для неподдерживаемых методов.

## СЕКЦИЯ 3: Contract Compliance Review (JSON формат vs Go DTOs)

### [LOW-001] client.updated: тип поля changes -- Структура vs ClientChanges (typed struct)

**Файл:** `module_kafka_integration.bsl` (функция `СформироватьДанныеКлиента`, строка 360)

**Описание:** В коде 1С `changes` формируется как `Структура` со смешанными типами: `is_strategic` (Булево), `allowed_deviation` (Строка), `manager_id` (Строка), `name` (Строка). В Go DTO `ClientChanges` использует typed struct с pointer-полями (`*bool`, `*string`). JSON-сериализация в 1С корректно обработает Булево как `true/false` и строки как строки -- контракт совпадает.

**Почему важно:** Риск низкий, контракт корректен. Но стоит отметить: если в будущем добавятся числовые поля в changes, нужно следить за типом (строка vs число).

**Рекомендация:** Добавить комментарий в код о типах полей changes для будущих разработчиков.

### [LOW-002] ls.created: перекопирование полей через цикл Для Каждого

**Файл:** `module_kafka_integration.bsl` (функция `СформироватьДанныеЛССоздан`, строки 385-387)

**Описание:** Функция копирует поля из входной структуры `ДанныеЛС` через цикл. Это означает, что формат JSON полностью зависит от вызывающего кода (поля не контролируются). Если вызывающий код передаст поле с неверным именем -- ошибка контракта не будет обнаружена.

**Почему важно:** Нарушение принципа explicit -- функция не контролирует, какие именно поля попадут в JSON. Контракт зависит от вызывающей стороны.

**Рекомендация:** Явно извлекать поля из `ДанныеЛС` по именам (как в других функциях), с валидацией обязательных полей (`external_id`, `client_id`, `total_amount`).

### [LOW-003] Дата occurred_at: захардкоженный offset +03:00

**Файл:** `module_kafka_integration.bsl` (функция `ФорматДатыISO`, строка 632)

**Описание:** Функция `ФорматДатыISO` всегда добавляет "+03:00" (МСК). По ТЗ это корректно. Однако если 1С-сервер работает в другом часовом поясе (например, UTC или Новосибирск), дата будет неверной -- ТекущаяДатаСеанса() вернет время сервера, а offset будет +03:00.

**Почему важно:** Для production-сервера в Москве -- без проблем. Но при миграции на сервер в другой ТЗ или при тестировании в облаке -- offset будет неверным.

**Рекомендация:** Добавить комментарий: "Предполагается, что сервер 1С в часовом поясе МСК". Для полной корректности можно использовать `МестноеВремяСеанса()` + определение реального offset через платформенные средства. Приоритет низкий.

## СЕКЦИЯ 4: Performance Review

### [LOW-004] Регламентное задание retry: интервал 60 секунд при 70 сеансах

**Файл:** `scheduled_job_retry.bsl`, `architecture_extension.md` (строка 78)

**Описание:** Регламентное задание `ekf_ОбработкаОчередиОтправки` запускается каждые 60 секунд. При 70 одновременных сеансах это создает дополнительную нагрузку. Однако фоновое задание легковесное: один запрос к РС + batch из 50 HTTP-вызовов.

**Почему важно:** 60 секунд -- агрессивный интервал. В memory (MEMORY.md) зафиксировано: "Timer interval: 60 sec creates load with 70 sessions, recommend 120 sec minimum". Однако для очереди событий скорость retry важнее.

**Рекомендация:** Оставить 60 секунд для первых попыток, но рассмотреть адаптивный интервал: если очередь пуста 3 раза подряд -- увеличить интервал до 120 секунд. При появлении записей -- вернуть 60.

### [LOW-005] Очистка expired записей и старых команд: удаление по одной записи в цикле

**Файл:** `scheduled_job_retry.bsl` (строки 131-135, 172-177)

**Описание:** Процедуры `ОчиститьОтправленныеСобытия` и `ОчиститьСтарыеКоманды` удаляют записи по одной через `МенеджерЗаписи.Удалить()` в цикле. При накоплении тысяч expired-записей это займет значительное время.

**Почему важно:** Очистка запускается в 03:00, когда нагрузка минимальна. Для типичных объемов (десятки записей) -- проблемы нет. Но при длительном сбое NiFi (несколько дней) может накопиться много expired-записей.

**Рекомендация:** Использовать `НаборЗаписей` с отбором для пакетного удаления:
```bsl
НаборЗаписей = РегистрыСведений.ekf_ОчередьОтправкиСобытий.СоздатьНаборЗаписей();
НаборЗаписей.Отбор.Статус.Установить("expired");
// + фильтр по дате
НаборЗаписей.Прочитать();
НаборЗаписей.Очистить();
НаборЗаписей.Записать();
```
Это атомарная операция, быстрее цикла.
